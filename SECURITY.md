# امنیت پروژه‌ی «مروارید فارم»

این سند مهم‌ترین موضوع امنیتی پروژه و نقشه‌ی راه رفع آن را توضیح می‌دهد.

---

## ۱) مشکل اصلی: استفاده از کلید `service_role` در سمت مرورگر (CRITICAL)

در حال حاضر برنامه یک کلاینت Supabase با کلید `service_role` می‌سازد
(`src/lib/supabase-admin.ts`) و تقریباً تمام عملیات داده (خواندن/نوشتن انبار،
کاربران، حواله‌ها، فرمول‌ها، تأمین‌کنندگان و …) از طریق همان کلاینت انجام می‌شود.

### چرا این یک خطر جدی است؟
- هر متغیری با پیشوند `VITE_` در زمان build داخل فایل خروجی (`dist/index.html`)
  قرار می‌گیرد. یعنی اگر `VITE_SUPABASE_SERVICE_ROLE_KEY` تنظیم شده باشد،
  این کلید برای **هر کاربری** که برنامه را باز کند، در مرورگر قابل استخراج است.
- کلید `service_role` تمام سیاست‌های RLS را دور می‌زند؛ یعنی هرکس آن را به‌دست آورد
  می‌تواند **کل دیتابیس** را بخواند، تغییر دهد یا حذف کند.
- محافظت‌های نقش‌محور سمت کلاینت (مثل `ProtectedRoute`) یک مرز امنیتی واقعی
  نیستند؛ فقط رابط کاربری را محدود می‌کنند و به‌راحتی دور زده می‌شوند.

### راهکار درست (نقشه‌ی راه)
این تغییر نیازمند دسترسی به پروژه‌ی Supabase شماست و در این مخزن قابل تست کامل نبود،
بنابراین به‌صورت نقشه‌ی راه ارائه شده است:

1. **در محیط production کلید `service_role` را تنظیم نکنید.** فقط `anon` بماند.
   کد `supabase-admin.ts` در نبود کلید سرویس، به کلید `anon` برمی‌گردد.
2. **سیاست‌های RLS را روی همه‌ی جدول‌ها فعال و تعریف کنید** تا کلاینت `anon`
   فقط به داده‌های مجاز هر نقش دسترسی داشته باشد (نمونه‌ی الگو در بخش ۴).
3. **عملیات مدیریتی** (ساخت/حذف/ریست کاربر و seed ادمین) را به
   **Supabase Edge Functions** منتقل کنید؛ کلید سرویس فقط داخل Edge Function
   (در سرور) قرار می‌گیرد و هرگز به مرورگر ارسال نمی‌شود.
4. **کلید سرویسی که قبلاً در بیلد قرار گرفته را در داشبورد Supabase باطل/چرخش
   (rotate) کنید**، چون باید فرض کنید لو رفته است.

## ۲) seed کردن ادمین در مرورگر

`src/utils/seedAdmin.ts` در `App.tsx` و در مرورگر اجرا می‌شود و برای ساخت ادمین به
کلید سرویس نیاز دارد. این کار باید به یک Edge Function یا اسکریپت سرور یک‌بارمصرف
منتقل شود. در حالت امن (بدون کلید سرویس) این تابع به‌صورت بی‌صدا خارج می‌شود و
ادمین باید از طریق سرور ساخته شود.

## ۳) موارد امنیتی دیگر

- **مدیریت لاگ:** لاگ‌های `console.*` با ابزار `src/utils/logger.ts` جایگزین شده‌اند
  که در production کاملاً ساکت است تا اطلاعات داخلی فاش نشود.
- **ذخیره‌سازی نشست:** نشست Supabase در `localStorage` نگه داشته می‌شود (رفتار
  پیش‌فرض supabase-js). توکن‌های دستی یا رمز عبور جایی ذخیره نمی‌شوند.
- **انقضای نشست سمت کلاینت:** تایمر انقضای نشست (`authStore`) صرفاً یک کنترل UX
  است و نباید به‌عنوان کنترل امنیتی اصلی در نظر گرفته شود؛ کنترل واقعی باید در
  سطح JWT/Supabase باشد.

## ۴) نمونه‌ی الگوی RLS (نیازمند بازبینی و تست روی دیتابیس شما)

> این فقط یک الگوی شروع است؛ پیش از اجرا روی production حتماً بازبینی و تست کنید.

```sql
-- فعال‌سازی RLS
alter table profiles enable row level security;
alter table inventory_transactions enable row level security;

-- هر کاربر پروفایل خودش را ببیند
create policy "read own profile"
  on profiles for select
  using ( auth.uid() = id );

-- ادمین همه‌ی پروفایل‌ها را ببیند (نقش از روی JWT خوانده می‌شود)
create policy "admin reads all profiles"
  on profiles for select
  using ( (auth.jwt() ->> 'role') = 'admin' );

-- کاربران فقط تراکنش‌های فارم خودشان را ببینند
create policy "read farm transactions"
  on inventory_transactions for select
  using (
    (auth.jwt() ->> 'role') = 'admin'
    or farm_id = (select farm_id from profiles where id = auth.uid())
  );
```
